Technical Documentation

Architecture, API reference, and internals for premiumize-explorer.

Architecture

premiumize-explorer is a single-process Qt6 desktop application written in C++20. It follows a signal/slot communication model: subsystems never call each other directly across boundaries — they emit signals that MainWindow listens to and routes.

Key classes

ClassFileRole
PremiumizeApisrc/api/PremiumizeApi.hppAll HTTP calls. Non-blocking — every call returns immediately and emits a signal when done.
TransferManagersrc/transfer/TransferManager.hppJob queue (max 2 concurrent). Owns UploadJob and DownloadJob instances.
PremiumizeModelsrc/model/PremiumizeModel.hppQAbstractListModel for the cloud pane. Flat list, one folder at a time.
UpEntryProxyModelsrc/model/UpEntryProxyModel.hppQIdentityProxyModel wrapping QFileSystemModel for the local pane.
FilePanesrc/ui/FilePane.hppReusable widget for both panes. Handles drag/drop, context menus, and toolbar actions.
TransferProgressWindowsrc/ui/TransferProgressWindow.hppNon-modal tool window tracking local upload/download jobs.
CloudTransfersWindowsrc/ui/CloudTransfersWindow.hppNon-modal tool window showing Premiumize server-side transfers. Polls every 5 s.
LogWindowsrc/ui/LogWindow.hppNon-modal tool window with a timestamped log of all API requests and responses.
BatchDownloadWizardsrc/ui/BatchDownloadWizard.hppModal QWizard — search cloud files by keyword, select with checkboxes, pick a destination, then download sequentially with dual progress bars and a per-file status list.
FormatHelperssrc/ui/FormatHelpers.hppInline helpers ui::formatBytes(qint64) and ui::formatDuration(qint64 ms) shared across all windows that display transfer metrics.
AppConfigsrc/config/AppConfig.hppSingleton QSettings facade. Persists API key, paths, geometry, theme preference.
MainWindowsrc/ui/MainWindow.hppRoot widget. Owns all subsystem instances and wires their signals together.

Data flow

PremiumizeApi ──signals──▶ MainWindow ──calls──▶ FilePane / PremiumizeModel ▲ │ │ ▼ TransferManager ◀──enqueue── FilePane (drag-drop / context menu) │ ├──▶ TransferProgressWindow (local job progress) └──▶ MainWindow::uploadFinished (triggers cloud pane refresh) PremiumizeApi::requestLogged ──▶ LogWindow PremiumizeApi::transferListReady ──▶ CloudTransfersWindow (polled every 5 s)

Authentication

Every request carries an Authorization: Bearer <api_key> header, set via PremiumizeApi::setApiKey(). The key is read from AppConfig on startup and can be changed at runtime via File → Change API Key…

Authorization: Bearer <your_api_key>

The API key is stored in ~/.config/premiumize-explorer/premiumize-explorer.ini under the [auth] group.

Endpoints

All methods are on api::PremiumizeApi (src/api/PremiumizeApi.hpp). Base URL: https://www.premiumize.me/api.

GET /folder/list

List the contents of a folder. Pass an empty string to list the root.

void listFolder(const QString& folderId = {});
ParameterWhereNotes
id optionalQuery stringFolder ID. Omit to list root.

Emits: folderListingReady(FolderListing)

POST /folder/create

Create a new cloud folder.

void createFolder(const QString& name, const QString& parentId = {});
ParameterWhereNotes
nameForm bodyNew folder name
parent_id optionalForm bodyParent folder ID; omit to create at root

Emits: folderCreated(newFolderId)

POST /folder/delete  ·  /item/delete

Delete a file or folder. The endpoint is chosen automatically based on the isFolder argument.

void deleteItem(const QString& id, bool isFolder);
ParameterWhereNotes
idForm bodyItem ID to delete

Emits: deleteFinished(success, errorMessage)

Single handler contract: deleteItem uses its own QNetworkReply::finished handler and does not call handleJsonReply. Never add a second connect to the same reply — both lambdas would fire on the same emission.
POST /folder/paste

Move files and/or folders into a target folder (used for drag-and-drop rearrangement).

void pasteItems(const QString& targetFolderId,
               const QStringList& fileIds,
               const QStringList& folderIds);
ParameterWhereNotes
idForm bodyDestination folder ID
files[]Form body (repeated)File IDs to move
folders[]Form body (repeated)Folder IDs to move

Emits: pasteFinished(success, errorMessage)

GET /folder/uploadinfo

Obtain a one-time upload token and CDN URL. Two variants exist:

// Shared-signal variant (for UI use)
void requestUploadInfo(const QString& targetFolderId = {});

// Private-reply variant — used by UploadJob to avoid token sharing
QNetworkReply* fetchUploadInfo(const QString& targetFolderId = {});
ParameterWhereNotes
id optionalQuery stringTarget folder ID; omit for root

Emits (shared variant): uploadInfoReady(UploadInfo)

Always use fetchUploadInfo inside UploadJob, not requestUploadInfo. The shared signal would deliver the same one-time token to all concurrently running upload jobs, causing all but the first to fail.
GET /account/info

Fetch quota usage. Called once on startup to populate the status bar.

void fetchAccountInfo();

Emits: accountInfoReady(AccountInfo)

GET /transfer/list

List all server-side transfers (torrents, URL/magnet downloads that Premiumize is processing). Called by CloudTransfersWindow every 5 s while the window is visible.

void fetchTransferList();

Emits: transferListReady(QList<CloudTransferEntry>)

GET /folder/search

Search all cloud files by keyword. Results are filtered to ItemType::File entries that have a valid link field. Used by BatchDownloadWizard.

void searchItems(const QString& query);
ParameterWhereNotes
qQuery stringSearch keywords

Emits: searchResultsReady(QList<FolderItem>)

GET /folder/list  (name resolution)

Resolve a folder's display name without showing its contents. The full content array in the response is discarded — only the top-level name field is read. Used by BatchDownloadWizard::SearchPage to append [FolderName] to search results.

void resolveFolderName(const QString& folderId);

Emits: folderNameResolved(folderId, folderName)

No lighter endpoint exists. Premiumize has no metadata-only endpoint; /folder/list is the closest option.
GET /folder/download

Fetch a pre-signed ZIP download URL for an entire cloud folder. Used by the cloud pane's context menu "Download" action for folder items that do not already have a link in the listing.

void fetchFolderDownloadLink(const QString& folderId);
ParameterWhereNotes
idQuery stringFolder ID

Emits: folderDownloadLinkReady(folderId, url) — always, on both success and failure. url is empty on error.

Own handler, not handleJsonReply. fetchFolderDownloadLink wires QNetworkReply::finished directly so it can emit folderDownloadLinkReady on the error path too — letting MainWindow always clean up its pendingFolderDownloads_ map entry. It additionally emits networkError on failure. The response field for the URL is "location".
GET <direct download URL>

Start a streaming download. Returns a raw QNetworkReply* that DownloadJob pipes directly to a QFile. No signal is emitted by PremiumizeApi; progress reporting goes through TransferManager.

QNetworkReply* startDownload(const QString& url);

Signals

All signals are on api::PremiumizeApi. They are always emitted on the main thread.

SIGNAL folderListingReady(FolderListing) — folder contents ready for display
SIGNAL folderCreated(QString newFolderId) — new folder ID from the server
SIGNAL deleteFinished(bool, QString) — success flag + optional error message
SIGNAL pasteFinished(bool, QString) — success flag + optional error message
SIGNAL uploadInfoReady(UploadInfo) — one-time token + CDN URL; do not use inside UploadJob
SIGNAL accountInfoReady(AccountInfo) — quota usage for the status bar
SIGNAL transferListReady(QList<CloudTransferEntry>) — current server-side transfer list
SIGNAL searchResultsReady(QList<FolderItem>) — file search results from /folder/search; consumed by BatchDownloadWizard
SIGNAL folderNameResolved(QString folderId, QString folderName) — display name for a folder ID; emitted by resolveFolderName, consumed by BatchDownloadWizard::SearchPage
SIGNAL folderDownloadLinkReady(QString folderId, QString url) — pre-signed ZIP URL for a folder; always emitted (empty url on error); consumed by MainWindow::pendingFolderDownloads_
SIGNAL networkError(QString) — emitted for any non-success response or network failure
SIGNAL requestLogged(QString) — one log line per request/response, consumed by LogWindow
handleJsonReply is a private helper that wires QNetworkReply::finished, reads the body, checks status == "success", and invokes a handler lambda on success only — it never calls the lambda on error. It also emits requestLogged with HTTP status, latency, and a truncated body.

Two endpoints use their own handlers instead: deleteItem avoids double-emitting networkError (its own handler emits deleteFinished(false, …) on error without also calling networkError). fetchFolderDownloadLink must always emit folderDownloadLinkReady — including on failure — so its caller can clean up the pending-requests map; it also emits networkError on error.

Upload flow

Uploads require two API calls because Premiumize uses a CDN for the actual file transfer.

  1. TransferManager::enqueueUpload creates an UploadJob and starts it when a slot is free.
  2. UploadJob::start calls api_->fetchUploadInfo(folderId), which returns a private QNetworkReply*.
  3. The reply is parsed to extract token and url.
  4. startUpload() builds a raw QByteArray multipart body and POSTs it to the CDN URL via a dedicated QNetworkAccessManager with Http2AllowedAttribute=false.
  5. On completion, TransferManager emits uploadFinished(folderId, success, error). MainWindow refreshes the cloud pane if the folder ID matches the one currently displayed.
Do not use QHttpMultiPart here. The energycdn.com CDN returns HTTP 500 for Qt's serialized multipart format. The body must be built manually as a QByteArray.

Field order matters: token must come before file in the multipart body — reversing the order also triggers HTTP 500.

Transfer manager

TransferManager (src/transfer/TransferManager.hpp) maintains a FIFO queue and runs at most MaxConcurrent = 2 jobs simultaneously.

Enqueueing

void enqueueUpload(const QStringList& localPaths, const QString& targetFolderId);
void enqueueDownload(const QString& remoteUrl, const QString& localDestPath,
                     qint64 expectedSize, const QString& itemName);

Each call pre-assigns a job ID (nextJobId_++), captures it in a factory lambda, and appends a pair<int, factory> to an internal std::deque. Jobs are not started until dispatchNext() pulls them; active jobs are stored as pair<int, QObject*> so IDs can be looked up at any time.

Cancellation

void cancelAll();            // cancel every queued and active job
void cancelJob(int jobId);   // cancel one specific job only

cancelJob is used by BatchDownloadWizard so cancelling a wizard download does not abort unrelated uploads. Both methods exit any loop over active_ before calling cancel() on the job to avoid iterator invalidation (see note below).

Signals emitted by TransferManager

SignalNotes
jobStarted(jobId, name, totalBytes)Consumed by TransferProgressWindow and BatchDownloadWizard
jobProgress(jobId, bytes, total, elapsedMs, etaMs, bytesPerSec)Fired continuously during transfer
jobFinished(jobId, success, error)Consumed by TransferProgressWindow and BatchDownloadWizard
uploadFinished(folderId, success, error)Consumed by MainWindow to auto-refresh the cloud pane
allFinished()Emitted when both the active list and queue are empty
cancelAll() / cancelJob() iterator safety: QNetworkReply::abort() emits finished synchronously, which chains into onJobFinished() → active_.erase(). cancelAll() iterates a snapshot copy of active_; cancelJob() exits its search loop and saves the target pointer before calling cancel(). Both approaches let the original vector be modified freely during cancellation.

Batch download wizard

BatchDownloadWizard (src/ui/BatchDownloadWizard.hpp) is a modal QWizard opened via File → Batch Download… (Ctrl+Shift+D). It downloads multiple cloud files one at a time via TransferManager.

Pages

PageClassDescription
SearchSearchPageText field + search button; results shown as a checkbox list with filename and size. Items without a link field are silently excluded. Next is enabled only when at least one item is checked.
DestinationDestinationPageShows a summary of selected files, a path field pre-filled from the local pane, and a Browse button. Next is enabled only when the path is a writable directory.
ProgressProgressPageTwo progress bars (current file: per-mille range 0–1000; overall: 0–n), a timer label (total / current file / ETA), a scrollable per-file status list, and a Cancel button.

Download flow

ProgressPage::initializePage() is called once when the wizard advances to the progress page. It reads the selected items from SearchPage and the destination from DestinationPage, then calls startNextFile(), which enqueues one download at a time. The active job ID is stored in currentJobId_; progress and finished signals from unrelated jobs are ignored.

Per-file status list

A QListWidget below the progress bars shows one row per file with a live status icon. Icons are sourced from style()->standardIcon(QStyle::SP_…):

StateIconWhen set

SuccessSP_DialogOkButtonon_jobFinished() when success == true
Error / No linkSP_MessageBoxCriticalon_jobFinished() on failure; startNextFile() for items without a download link (also scrolls)

Cancel safety

Three entry points all converge on ProgressPage::cancelBatch():

  • The Cancel button
  • Pressing Escape or clicking the window × (handled by BatchDownloadWizard::reject())

cancelBatch() sets cancelled_ = true first, then calls manager_->cancelJob(currentJobId_). Because cancelJob fires jobFinished synchronously, on_jobFinished guards with if (cancelled_) return to skip the re-entrant call — preventing the total-progress bar from advancing one extra step.

Cloud transfers window

CloudTransfersWindow (src/ui/CloudTransfersWindow.hpp) shows server-side Premiumize transfers — torrents and URL/magnet downloads that Premiumize is processing for you, as opposed to local upload/download jobs managed by TransferManager.

Polling

A QTimer fires PremiumizeApi::fetchTransferList() every 5 seconds (kPollIntervalMs). The timer starts in showEvent and stops in hideEvent, so no requests are made while the window is hidden.

Row lifecycle

On each transferListReady signal the window rebuilds its row list from scratch. Each TransferRow tracks both its container widget and its preceding separator QFrame; both are deleted during cleanup. The separator is not a Qt child of the container — it must be deleted explicitly.

API log window

LogWindow (src/ui/LogWindow.hpp) is a non-modal Qt::Tool window fed by the PremiumizeApi::requestLogged signal.

Each entry is a single timestamped line in the format:

[HH:mm:ss.zzz] → GET https://www.premiumize.me/api/folder/list
[HH:mm:ss.zzz] ← 200 (143ms) {"status":"success","content":[...]}
[HH:mm:ss.zzz] ← ERR (50ms) Network timeout

The log can be saved to a plain-text file via the Save to File… button, or cleared with Clear.

File panes

Both the local and cloud panes are instances of FilePane (src/ui/FilePane.hpp), distinguished by PaneType::Local and PaneType::Cloud.

Local pane

Backed by QFileSystemModel wrapped in UpEntryProxyModel, which injects the virtual ↑ Up row. The proxy stores viewRoot_ as a plain QModelIndex (not QPersistentModelIndex) because endResetModel() immediately invalidates persistent indices.

Cloud pane

Backed by PremiumizeModel, a flat QAbstractListModel showing one folder at a time. The ↑ Up row is injected internally by the model and derived from currentFolderId_ / parentFolderId_ — there is no separate boolean flag that could drift out of sync.

Context menu download (cloud pane)

The "Download" action supports multi-selection. At menu-build time the target list is computed: if the right-clicked item is in the current selection, the full selection is used; otherwise only the right-clicked item is targeted. The action is only added to the menu when at least one target has a link or is a folder.

Target typeAction
File with linkEmits downloadRequested immediately
Folder with linkEmits downloadRequested with .zip suffix
Folder without linkEmits folderDownloadRequested; MainWindow calls fetchFolderDownloadLink and enqueues the transfer when folderDownloadLinkReady fires
File without linkSilently skipped; menu item hidden if all targets are linkless non-folders
Model index safety: PremiumizeModel::itemAt(row) asserts and throws on out-of-range access. All call sites in FilePane guard with row >= 0 && row < cloudModel_->rowCount() before calling it — the model can be reset asynchronously by an in-flight listFolder reply.

Drag & drop

DirectionMIME typeSignal emitted
Local → Cloudtext/uri-listFilePane::uploadRequested(paths, folderId)
Cloud → Localapplication/x-premiumize-itemsFilePane::downloadRequested(url, dest, size, name)

The application/x-premiumize-items payload is a JSON array of objects with keys id, name, isFolder, link, size.

DragOnly mode is critical. Both list views use setDragDropMode(DragOnly), not DragDrop. With DragDrop, the viewport calls setAcceptDrops(true) and intercepts drops through the model's canDropMimeData/dropMimeData chain, which has no signal-emitting implementation. DragOnly keeps acceptDrops false on the viewport, so Qt delivers drop events directly to the FilePane parent widget.
